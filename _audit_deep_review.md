> **목적**: 정적 스캔·회귀·property·fuzzing·pairwise가 **원리적으로 못 잡는 "의미 로직
> 버그"**(그럴듯하지만 틀린 값·죽은 경로·잘못된 배정)를 에이전트 팬아웃으로 발굴한다.
> 자동 그물(도구)이 구조·경계·조합을 보는 동안, 이 절차는 **사람 수준 코드 독해**를 병렬화한다.
> 정본 트리거·정책: `CLAUDE.md` 종합 감사 ①코드·로직 영역. 이 파일은 그 실행 레시피.

## 언제 (트리거)
- **major 블록 전환 직전**(주 트리거) — 누적 diff 전체를 통합 상태에서.
- 사용자가 "깊은 감사/전체 점검" 요청 시.
- 누적 위험이 크다고 판단될 때(Claude 제안 → 사용자 승인).

> ⚠ 에이전트 spawn은 **사용자 명시 승인 후에만**. 이 문서는 승인 시 쓸 레시피이지 자동 실행이 아니다.

## 각 에이전트 공통 프롬프트 (read-only, 수정은 취합 후 부모가)
```
<파일>을 정밀 정독해 로직 버그·구현 미비를 찾아라. 각 발견은 file·line·요약·
구체적 실패 시나리오(어떤 입력/상태 → 잘못된 출력/크래시)로. 다음을 특히 본다:
- 전역 상태 오염(전역 DB를 .copy() 없이 mutate / cfg 미복사)
- 0나눗셈·NaN/Inf·확률 clamp 누락·단위 혼동(km↔m·ms↔s)
- 상태머신 무동작(도달 불가 분기·죽은 토글·장식 상태)
- 부모 계약 위반(상속이 시그니처·훅 기본동작 변경)
- MC 3경로 정합(신규 stats 키 누락)·회귀 골든 사각(안 밟히는 경로)
- 순회 필터 누락(잘못된 객체가 잘못된 경로로 — 예: CAP기가 대잠 순회에)
- 죽은 코드(호출처 0)·미소비 필드
자동 그물(정적38·회귀·property·fuzz·pairwise)이 이미 PASS인 것은 baseline으로 두고,
그것들이 원리적으로 못 잡는 의미 로직에 집중하라.
```

## 취합·검증·수정 절차
1. 4개 결과를 **심각도순**(크래시 > 지표왜곡 > 위생) 취합, 중복 제거.
2. 각 발견을 **부모가 직접 재현·검증**(grep·드라이버·traceback) — 에이전트 주장 그대로 믿지 않음.
3. 확정 버그는 **저위험→고위험 순** 수정, 각 단계 후 회귀. 대량삭제는 [[feedback-bulk-line-delete]].
4. **놓친 경로는 회귀 골든에 케이스 추가**(재발 시 자동 차단) — 이번 CAP+잠수함 선례.
5. 발견 성격이 갈리면(설계 판단·비가역) 사용자에게.
