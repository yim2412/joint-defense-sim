## 0. 결론 먼저 — 지금까지 확정된 것

- **부분적으로 로컬 가능하다.** 단 **`app_main.py` 분할이 전제**다(아래 §2).

## 1-b. ✅ 스택 구축 완료 — 실측 결과 (2026-07-17 세션)

**§6 ①~④ 전부 완료. 함정 2개 다 실측으로 확인됐고 둘 다 해소.**

| 항목 | 결과 |
|------|------|
| Ollama 데몬 | 기동 OK · CUDA 인식(compute 8.6) · 여유 **8.9GiB** |
| 모델 | `qwen2.5-coder:7b` (Q4_K_M · ctx 32768) · **`14b`가 3주 전부터 이미 있음**(9.0GB) |

| 연결 검증 | `aider --model ollama/qwen2.5-coder:7b` → 정상 응답 |

### 계획서 예측이 빗나간 4곳 (다음 세션이 같은 가정을 반복하지 않도록)
1. **winget이 이 PC에 없다.** AppInstaller 패키지는 있으나 `winget.exe` 실행 별칭이 없다.
   → **uv로 전환**(사용자 결정). `irm https://astral.sh/uv/install.ps1 | iex` → `~/.local/bin`.
   시스템 전역 오염 없음. **uv는 3초에 3.12+aider를 끝냈다** — winget보다 나은 경로였다.
2. **Python 3.14 함정은 진짜였고, 증상이 예상과 달랐다.** `pip install aider-chat`이 "미지원"
   에러를 내지 않는다 — pip이 **2023년 릴리스로 조용히 후퇴**(`openai==0.27.6`·`aiohttp==3.8.4`)한 뒤
   그게 빌드 실패한다(`Cannot import 'setuptools.build_meta'`). **에러 메시지가 원인을 안 가리킨다.**
3. **7B 기본 태그는 Q5가 아니라 Q4_K_M.** §3 표의 "7B Q5"는 틀림.
4. **Q8 KV 양자화는 필요 없었다**(위 표) — f16으로도 32k가 VRAM에 들어간다.

### 환경 재현 (새 세션·재부팅 후)
```bash
export PATH="/c/Users/준/.local/bin:$PATH"     # uv·aider
export OLLAMA_API_BASE=http://127.0.0.1:11434
export PYTHONIOENCODING=utf-8                   # aider의 cp949 크래시 회피(§1-c)
ollama serve &                                  # 데몬은 재부팅 시 재기동 필요
aider --model ollama/qwen2.5-coder:14b          # ★ 7B 아님 — 7B는 실측 탈락(§1-c)
```
- `num_ctx`는 **`.aider.model.settings.yml`**(repo 루트·커밋됨)이 담당. 7b·14b 둘 다 32768.
  Modelfile 커스텀 태그를 만들 필요 없다.
- aider 세션 산출물·serve 로그는 `.gitignore`에 등록됨(설정 yml만 커밋).
- **검증은 로그로**: `grep "n_ctx" _ollama_serve.log` → `n_ctx = 32768`이어야 한다. 추측 금지.

## 1-c. ✅ 3단계 실전 테스트 결과 (2026-07-17 · 실측)

**결론: 로컬은 성립한다 — 단 `14B` + `작은 파일`의 조합에서만. 7B는 탈락.**

| 단계 | 모델 | 대상 | 결과 | 게이트 채점 |
|------|------|------|------|-------------|
| 2 | **7B** | `engine_joint.py` (6k·"✅여유"로 분류했던 파일) | ❌ **실패** | 3회 재시도 전부 `SearchReplaceNoExactMatch` · 파일 변경 0 |
| 2 | **14B** | 동일 | ✅ **합격** | py_compile PASS · 정적 51/51 PASS · **회귀 38×29 PASS** |
| 3 | **14B** | `app_main.py` (238k 실측) | ❌ **실패**(예상대로) | 컨텍스트 7배 초과 · 파일 변경 0(무손상) |

### 🔴 §2의 핵심 주장이 절반 틀렸다 — "병목은 지능이 아니라 파일 크기"
**둘 다 병목이었다.**
- **7B는 6k 파일에서 실패했다.** 토큰은 **14k sent**로 32k 안에 여유롭게 들어갔다 —
  **컨텍스트 부족이 아니다.** 7B가 SEARCH 블록을 **환각**했다(실제론 `joint_dmg_share` 다음에
  `joint_share_air/navy/army`가 있는데 `joint_fire_log`가 바로 온다고 지어냄).
  **눈앞의 파일을 그대로 옮겨 적지 못한다** = 순수 지능 문제. 파일을 아무리 쪼개도 7B로는 안 된다.
- **§3의 "7B 1순위 추천"·"7B가 14B보다 낫다(컨텍스트 때문)"는 틀렸다.** 둘 다 ctx 32768로 같다.
  14B는 **1회 시도로 정확한 diff**를 냈다(들여쓰기·정렬까지 프로젝트 스타일 준수).
- **§2의 파일 크기 주장은 맞았다.** 3단계에서 **aider 자신이** 답했다:
  > `Your estimated chat context of 237,946 tokens exceeds the 32,768 token limit`
  > `- Break your code into smaller files`

  **⇒ `app_main.py` 분할은 여전히 전제다.** 다만 이유가 하나 늘었다 — 분할만으로는 부족하고
  **모델도 14B 이상이어야 한다.**

### 속도 실측 — dense 오프로드 손해는 §3 예측대로였다
| 조건 | 속도 | 해석 |
|------|------|------|
| 14B · 12k ctx (2단계) | **~10초에 완료** | 실용 속도. VRAM 안 |
| 14B · 238k ctx (3단계) | **2.34 t/s** | VRAM 초과 → RAM 오프로드. §3의 "dense 10배 손해" 확인 |

**⇒ 파일이 작으면 14B는 충분히 빠르다.** 오프로드는 큰 파일에서만 터진다 = 분할이 속도까지 해결.
**MoE 30B-A3B 실측은 아직 미실시**(14B로 이미 성립해 급하지 않음. §3 승부처는 열려 있음).

### 부수 발견 (환경 — 다음 세션이 또 밟지 않도록)
- **aider가 컨텍스트 초과를 막지 않는다.** 경고만 하고 **그대로 보낸다** → 모델이 잘린 맥락으로
  편집을 시도할 수 있다. 다행히 이번엔 형식 위반으로 무산됐고 **`app_main.py`는 변경 0줄**이었지만,
  **큰 파일에 로컬을 붙이면 조용한 오편집 위험이 있다.** 게이트가 마지막 방어선인 이유.
- **Git Bash에서 cp949 크래시**: aider가 `•`(U+2022) 출력 시 `UnicodeEncodeError`. 진행은 되나
  로그가 깨진다. `PYTHONIOENCODING=utf-8` 또는 cmd.exe 사용 검토.
- **aider 버그**: SEARCH 실패 메시지를 파일명으로 오인해 `git add`를 시도 → repo 루트에
  `SearchReplaceNoExactMatch`라는 쓰레기 파일 생성(삭제함).

## 1-d. ✅ MoE 30B-A3B 실측 — §3 "승부처" 종결 (2026-07-17)

**결론: 30B-A3B가 이 PC의 최적 모델이다. 하지만 `app_main.py`는 여전히 못 만진다 — 이유가 바뀌었다.**

`ollama pull qwen3-coder:30b` · `architecture qwen3moe` · 30.5B(활성 3.3B) · **ctx 262144** · Q4_K_M · 18GB

| 층 | 과제 | 결과 |
|----|------|------|
| **A** | `joint_total_fired` 키 추가(14B가 통과한 것) | ✅ **1회·55초** · py_compile PASS. **배치가 14B보다 정확**(`navy_fired`/`army_fired` 바로 뒤) |
| **B** | 해군 재고 소진 표면화 — **명세 없이 패턴 유추** | ⚠ **게이트 3종 PASS인데 버그 2개**(아래) |
| **C** | `app_main.py`(238k) 편집 | ❌ **실패** — 단, **컨텍스트 초과가 아니다**(아래) |

### 🔴 §3의 MoE 가설은 맞았다 — 웹 권고가 이 PC에서는 틀렸다
- `ollama ps`: **`20GB · 60%/40% CPU/GPU`인데 생성 35.6 t/s.** 14B가 오프로드됐을 때 **2.34 t/s**였다
  → **15배**. §3의 "MoE는 토큰당 활성 3B만 읽어 오프로드 손해가 작다"가 실측으로 확인.
- 2026-07 웹 권고("Ollama의 MoE GPU 활용 이슈 때문에 dense를 써라")는 **이 PC·이 버전에선 반대**였다.
  dense 14B가 오프로드되면 죽고, MoE 30B는 60% CPU에서도 살아 있다.
- **⇒ 후보 3종 실측 완료. 승자는 MoE 30B.** §3 표의 "7B 1순위"는 폐기(§1-c).

### 🔴 C층 실패의 진짜 원인 — 컨텍스트가 아니라 **prefill 시간 > 클라이언트 타임아웃**
```
qwen3-coder:30b   45 GB   82%/18% CPU/GPU   CONTEXT 262144   ← 238k가 실제로 들어감
prompt processing, progress = 0.99, t = 597.90 s / 130 t/s   ← 99%에서
litellm.Timeout: Connection timed out after 600.0 seconds     ← 2.1초 차이로 잘림
```
- **`app_main.py` 238k는 262144 컨텍스트에 들어간다.** 14B가 낸 `exceeds the token limit` 경고가
  **30B에선 안 나온다.** ⇒ **§2의 "286k는 컨텍스트를 늘려 해결할 수 없다·128k 모델도 안 들어간다"는 틀렸다.**
- **하지만 못 쓴다**: prefill 338→130 t/s로 감속(어텐션이 누적 토큰을 다 봄) → **10분 초과** →
  **aider의 600초 HTTP 타임아웃**이 요청을 죽인다. 재시도해도 캐시가 조금 밀어올릴 뿐 같은 벽.
  **prefill이 끝나도 생성 예산이 0**이라 어차피 실패. 타임아웃을 늘려도 **매 요청 10분+는 실용성 0**.
- **⇒ 분할 결론은 유지되나 근거가 바뀐다: "물리적으로 불가능"이 아니라 "시간이 실용 밖".**
  분할하면 편집 가능성과 **속도**를 동시에 얻는다(§1-c의 12k=10초 사례).

### 🔴 B층이 이번 테스트의 최대 소득 — **게이트는 "안 망가뜨림"만 증명한다**
30B의 B층 답: `army_fire_unused` 패턴을 **스스로 찾아** `navy_fire_unused`를 만들고, 네이밍
(`joint_navy_unused`)·주석 스타일·`__init__`/`_navy_effort`/`summary()` **4곳을 일관 수정**했다.
**py_compile·정적 51/51·회귀 38×29 전부 PASS.** 그런데 **버그가 2개 있다**:
1. **플래그가 영구 True로 박힌다.** `if not any_navy_fire: self.navy_fire_unused = True` — True만
   세우고 False로 되돌리는 경로가 없다. `_navy_effort`는 **매 틱** 호출되므로, 첫 틱에 재고가 없으면
   나중에 재보급받아 쏴도 영원히 True. 원본 `army_fire_unused`는 매번 **재할당**해 True/False가 오간다
   → **패턴을 절반만 베꼈다.**
2. **"재고 소진"과 "함정 부재"를 구분하지 않는다.** 요청은 "재고가 바닥나면"인데 함정 0척·전원 수리 중도
   True가 된다 → 바로 위 주석("재고 바닥나서 미참여")이 **코드와 다른 말을 한다.**

**⇒ §4("약한 모델 + 튼튼한 관문 > 강한 모델 + 관문 없음")의 한계가 실물로 나왔다.**
관문은 **"기존을 안 깨뜨렸다"**를 증명하지 **"새로 맞게 만들었다"**를 증명하지 않는다.
하필 `joint_fires`는 짝 3중 ON에서만 생성돼 회귀 골든 38케이스에 안 걸린다 = 이 버그는 원리상 통과.

## 2. 🔴 핵심 발견 — 파일 크기가 편집 가능 여부를 정한다

> ⚠ **2026-07-17 실측으로 절반 정정됨 — §1-c를 먼저 읽을 것.**
> ⚠ **§1-d 추가 정정: "286k는 컨텍스트로 해결 불가"는 틀렸다.** 30B(262144)엔 들어간다.
> 못 쓰는 이유는 **prefill 10분 > aider 600초 타임아웃**이다(물리적 불가 아님). 파일 크기는 진짜 병목이 맞지만
> **유일한 병목이 아니다.** 7B는 6k 파일에서도 실패했다(지능). 아래 "로컬 가능" 열은
> **14B 기준으로 다시 읽어야 한다** — 7B 기준 판정은 낙관적이었다.

aider의 편집 단위는 **파일**이다(함수만 넣을 수 없다). 그래서 파일 토큰 수 = 편집 가능 여부.

| 파일 | 줄 | 대략 토큰 | 로컬(7B·16k) |

## 3. 모델 선택 — RAM 오프로드는 MoE에서만 성립

토큰 생성은 **메모리 대역폭** 바운드(매 토큰마다 활성 가중치 전부를 읽는다).

| | 대역폭 |
|---|---|
| RTX 3080 VRAM | **~760 GB/s** |
| DDR5 듀얼채널(7500F) | **~70 GB/s** |

**약 10배 차이** → RAM으로 내린 부분은 그만큼 느려진다. 용량(64GB)은 넉넉한데 **파이프가 좁다**.

- **Dense 모델**(Qwen2.5-Coder 14B 등): 매 토큰 8.5GB 전부를 읽음 → 오프로드하면 **10배 손해**.
  **RAM 오프로드로 dense 14B 굴리는 건 비효율.**
- **MoE 모델**(Qwen3-Coder-30B-A3B 등): 총 18GB지만 **토큰당 활성 3B(~1.8GB)뿐** → 읽는 양이
  1/10이라 오프로드 손해가 작다. **오프로드가 성립하는 유일한 구조. 64GB RAM이 값하는 지점.**
- ⚠ **두 번째 함정 — prefill**: 입력 처리는 대역폭이 아니라 **연산 바운드**라 CPU가 특히 약하다.
  aider는 **매 요청마다 컨텍스트를 재전송**하므로, 16k prefill이 느리면 생성이 빨라도 매 턴이 느리다.
  6코어라 이게 실질 병목일 수 있다.

### 후보 3종 — ✅ 전부 실측 완료(2026-07-17). 승자 = **MoE 30B**. 상세 §1-c·§1-d

## 4. ⭐ 채점은 게이트로 — 이 repo의 최대 자산

대부분의 프로젝트는 "이 로컬 모델이 쓸 만한가"를 **감으로** 답한다. 이 repo는 **실측으로** 답한다:

```
py_compile → audit_static_scan (51항목) → audit_verify_regression (38×29) → pre-commit 통과 여부
```

**"괜찮아 보인다"가 아니라 "커밋이 되느냐"로 판정한다.** 약한 모델이 엔진을 잘못 건드리면
**골든 불일치로 커밋 자체가 막힌다.** 이게 로컬 전환에서 살아남는 이유다 —
**약한 모델 + 튼튼한 관문 > 강한 모델 + 관문 없음.**

> 2026-07-17 세션에서 이 게이트가 Claude의 실수를 **3번** 잡았다(README 단계 버전 stale,
> patch_queue 버전 stale, README 파일구조표 미기재). 7B에겐 더 자주 그럴 것이다 = 설계대로.

## 7. 정직하게 — 무엇이 바뀌고 무엇이 안 바뀌나

**바뀐다**: 오늘(2026-07-17) 같은 세션은 재현되지 않는다. `enable_sead`가 적 IADS도 켠다는 걸

**안 바뀐다**: 그걸 잡는 건 원래도 모델이 아니었다. 오늘 Claude를 세 번 멈춰 세운 건
사용자였다("3분 걸렸다"→실제 90초 / "구조적으로 불가능"→이미 해결돼 있었음 /
"기존엔 잘 되던데"→도구 미사용). **로컬 전환에서 바뀌는 건 "누가 타이핑하나"지
"누가 판단하나"가 아니다.**

**역할 재정의**: 로컬 모델은 *판단하는 조수*가 아니라 **타이핑을 대신하는 도구**.
"이상한데?"라고 멈추는 건 사용자. 게이트가 그 판단을 뒷받침한다.
