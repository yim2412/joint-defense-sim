> 정본 계획이다. 다음 세션은 대화 기억이 0이므로 **이 파일만 읽고 이어갈 수 있게** 쓴다.

## 0. 결론 먼저 — 지금까지 확정된 것

- **부분적으로 로컬 가능하다.** 단 **`app_main.py` 분할이 전제**다(아래 §2).

### 계획서 예측이 빗나간 4곳 (다음 세션이 같은 가정을 반복하지 않도록)
1. **winget이 이 PC에 없다.** AppInstaller 패키지는 있으나 `winget.exe` 실행 별칭이 없다.
   → **uv로 전환**(사용자 결정). `irm https://astral.sh/uv/install.ps1 | iex` → `~/.local/bin`.
   시스템 전역 오염 없음. **uv는 3초에 3.12+aider를 끝냈다** — winget보다 나은 경로였다.
2. **Python 3.14 함정은 진짜였고, 증상이 예상과 달랐다.** `pip install aider-chat`이 "미지원"
   에러를 내지 않는다 — pip이 **2023년 릴리스로 조용히 후퇴**(`openai==0.27.6`·`aiohttp==3.8.4`)한 뒤
   그게 빌드 실패한다(`Cannot import 'setuptools.build_meta'`). **에러 메시지가 원인을 안 가리킨다.**
3. **7B 기본 태그는 Q5가 아니라 Q4_K_M.** §3 표의 "7B Q5"는 틀림.
4. **Q8 KV 양자화는 필요 없었다**(위 표) — f16으로도 32k가 VRAM에 들어간다.

### 환경 재현 (새 세션·재부팅 후)

- aider 세션 산출물·serve 로그는 `.gitignore`에 등록됨(설정 yml만 커밋).

## 2. 🔴 핵심 발견 — 파일 크기가 편집 가능 여부를 정한다

aider의 편집 단위는 **파일**이다(함수만 넣을 수 없다). 그래서 파일 토큰 수 = 편집 가능 여부.

| 파일 | 줄 | 대략 토큰 | 로컬(7B·16k) |
|------|-----|-----------|--------------|
| `app_main.py` | 12,807 | **286k** | ❌ **불가** |
| `engine_combat.py` | 8,586 | 183k | ❌ 불가 |
| `db_specsheet.py` | 3,516 | 72k | ❌ 불가 |
| `engine_core.py` | 2,563 | 61k | ❌ 불가 |
| `engine_campaign.py` | 1,119 | 27k | 🔶 32k 확보 시 |
| `engine_army.py` | 576 | 13k | ✅ 가능 |
| `engine_airforce.py` | 488 | 11k | ✅ 가능 |
| `engine_joint.py` | 243 | 6k | ✅ 여유 |
| `CLAUDE.md` | 523 | 18k | (규칙 — §5 참조) |

**⇒ v19~v21에 새로 만든 엔진 층은 이미 로컬 편집 가능. 구형 덩어리는 불가.**

### 왜 `app_main.py` 분할이 "선택"이 아닌가
- **`enable_xxx` 3종 세트 규칙이 `app_main.py`를 반드시 건드리게 만든다**
  (체크박스 + cfg 빌드 + cfg 로드). 즉 **신기능을 하나라도 추가하려면 286k 파일을 열어야 한다.**
- **`SCENARIO_LIBRARY`도 `app_main.py:6234`에 있다** → **v21.3(통합 시나리오)은 난이도가
  낮은데도 로컬로 못 한다.** (2026-07-17 세션에서 "v21.3은 로컬 몫"이라 분배했다가 **철회** —
  난이도만 보고 파일 크기를 안 봤다.)
- 286k는 컨텍스트를 늘려 해결할 수 없다. 128k 모델도 안 들어간다.
- **탭 단위로 쪼개면 조각당 1~2천 줄 = 로컬 편집권**에 들어온다.
- **이 리팩터는 안전하게 할 수 있다** — 회귀 골든 38×29 + round-trip(토글 61개) + GUI 스모크가
  동작 보존을 증명한다. 쪼갠 뒤 골든이 그대로면 성공.

## 3. 모델 선택 — RAM 오프로드는 MoE에서만 성립

토큰 생성은 **메모리 대역폭** 바운드(매 토큰마다 활성 가중치 전부를 읽는다).

| | 대역폭 |
|---|---|
| RTX 3080 VRAM | **~760 GB/s** |
| DDR5 듀얼채널(7500F) | **~70 GB/s** |

**약 10배 차이** → RAM으로 내린 부분은 그만큼 느려진다. 용량(64GB)은 넉넉한데 **파이프가 좁다**.

- **Dense 모델**(Qwen2.5-Coder 14B 등): 매 토큰 8.5GB 전부를 읽음 → 오프로드하면 **10배 손해**.
  **RAM 오프로드로 dense 14B 굴리는 건 비효율.**
- **MoE 모델**(Qwen3-Coder-30B-A3B 등): 총 18GB지만 **토큰당 활성 3B(~1.8GB)뿐** → 읽는 양이
  1/10이라 오프로드 손해가 작다. **오프로드가 성립하는 유일한 구조. 64GB RAM이 값하는 지점.**
- ⚠ **두 번째 함정 — prefill**: 입력 처리는 대역폭이 아니라 **연산 바운드**라 CPU가 특히 약하다.
  aider는 **매 요청마다 컨텍스트를 재전송**하므로, 16k prefill이 느리면 생성이 빨라도 매 턴이 느리다.
  6코어라 이게 실질 병목일 수 있다.

### 후보 3종 (전부 실측할 것 — 위 숫자는 이론값이지 이 PC의 사실이 아니다)
| 후보 | 구성 | 기대 |
|------|------|------|
| Qwen2.5-Coder **7B** Q5 | 전부 VRAM | 기준선 · 가장 빠름 · **1순위 추천** |
| Qwen2.5-Coder **14B** Q4 | 일부 RAM | 이론상 큰 손해 — 확인용 |
| **MoE 30B-A3B** Q4 | 대부분 RAM | **오프로드 유효 케이스 — 승부처** |

> ⚠ 지식 기준일(2026-01) 이후 더 나은 모델이 나왔을 수 있다. 최신 코딩 벤치마크를 확인하고
> 후보를 갱신할 것. **"7B가 14B보다 낫다"는 건 컨텍스트 때문** — 지능보다 컨텍스트가 중요하다.

## 4. ⭐ 채점은 게이트로 — 이 repo의 최대 자산

대부분의 프로젝트는 "이 로컬 모델이 쓸 만한가"를 **감으로** 답한다. 이 repo는 **실측으로** 답한다:

```
py_compile → audit_static_scan (51항목) → audit_verify_regression (38×29) → pre-commit 통과 여부
```

**"괜찮아 보인다"가 아니라 "커밋이 되느냐"로 판정한다.** 약한 모델이 엔진을 잘못 건드리면
**골든 불일치로 커밋 자체가 막힌다.** 이게 로컬 전환에서 살아남는 이유다 —
**약한 모델 + 튼튼한 관문 > 강한 모델 + 관문 없음.**

> 2026-07-17 세션에서 이 게이트가 Claude의 실수를 **3번** 잡았다(README 단계 버전 stale,
> patch_queue 버전 stale, README 파일구조표 미기재). 7B에겐 더 자주 그럴 것이다 = 설계대로.

## 6. 스택 검증 절차 (1순위 — 이대로 실행)

```bash

#    확인: 컨텍스트가 실제로 반영됐는지 로그로 검증할 것(추측 금지)

# ③ venv (Python 3.14 함정 — aider가 3.11/3.12 요구할 수 있음)
```

### 3단계 실전 테스트 — 속도가 아니라 **게이트 통과**로 채점
| 단계 | 과제 | 파일 | 합격 기준 | 예상 |
|------|------|------|-----------|------|
| **1** | changelog 문구 정리 | `app_changelog.json` | 정적 스캔 통과 | 됨 |
| **2** | `engine_joint.py` 실제 수정 1건 | 6k — 여유 | **회귀 38×29 PASS** | **분수령** |
| **3** | v21.3 프리셋 1종 | `app_main.py` 286k | — | **실패 예상 = 그게 결과** |

- **2단계가 진짜 시험.** 통과하면 **엔진 층 작업이 로컬로 열린다.**
- **3단계는 실패하라고 넣는다** — "분할이 정말 필요한가"를 이론이 아니라 **이 PC의 사실로** 확인.
- 측정 지표: 생성 tok/s · 16k prefill 시간 · **실과제 성공 여부**(← 제일 중요).
  30B가 12 tok/s라도 **한 번에 맞히면** 40 tok/s로 세 번 틀리는 7B보다 빠르다.

### ⚠ 테스트 오염 금지
**Claude가 프롬프트를 다듬고 실패를 고쳐주면 결과가 무의미해진다** — "로컬로 된다"는 잘못된
결론이 나온다. Claude 역할은 **환경 구축까지**. 과제 지시·판정은 사용자와 게이트가 한다
(Claude가 없을 때의 조건을 그대로 재현해야 하므로).

## 7. 정직하게 — 무엇이 바뀌고 무엇이 안 바뀌나

**바뀐다**: 오늘(2026-07-17) 같은 세션은 재현되지 않는다. `enable_sead`가 적 IADS도 켠다는 걸

**안 바뀐다**: 그걸 잡는 건 원래도 모델이 아니었다. 오늘 Claude를 세 번 멈춰 세운 건
사용자였다("3분 걸렸다"→실제 90초 / "구조적으로 불가능"→이미 해결돼 있었음 /
"기존엔 잘 되던데"→도구 미사용). **로컬 전환에서 바뀌는 건 "누가 타이핑하나"지
"누가 판단하나"가 아니다.**

**역할 재정의**: 로컬 모델은 *판단하는 조수*가 아니라 **타이핑을 대신하는 도구**.
"이상한데?"라고 멈추는 건 사용자. 게이트가 그 판단을 뒷받침한다.

## 8. 참고 — 결제 없는 대안 (한 번만 기록)

무료 티어(Gemini AI Studio 등)는 **결제 없이** 쓸 수 있고 컨텍스트가 커서
`app_main.py` 286k 문제에 훨씬 강하다. "완전 로컬"의 이유가 **오프라인·프라이버시**면 그대로 가고,
**비용**이면 병행이 실익이 크다. (사용자에게 한 번 제시함 — 재권유하지 말 것.)
